Qwen 3.5「small」オープンモデル：長文コンテキスト＋マルチモーダルのオンデバイス実行が現実に

  • Qwen3.5-0.8B / 2B / 4B / 9B（Base + Instruct）リリース：Alibabaが「より少ない計算でより高い知能」を掲げたコンパクトシリーズを発表しました。ネイティブマルチモーダルとスケーリングされたRLを搭載し、エッジ＋軽量エージェント展開を明確にターゲットとしています（Alibaba_Qwen）。コミュニティでは262Kのネイティブコンテキスト（最大1Mまで拡張可能）や、ツイートで報告された競争力のあるスコア（例：「82.5 MMLU-Pro」、「78.4 MMMU」、「97.2 CountBench」）が注目されていますが、これらはベンダーや二次情報による主張であるため、モデルカードを確認することが推奨されます（kimmonismus）。
  • アーキテクチャに関するコメントが浮上：複数のツイートが、Qwenがハイブリッド／非正統的アテンションへ移行していることを指摘しています。3.5では以前のQwen3の「Thinking vs Instruct」分離から「ハイブリッドモデル」に回帰しているとのことです（nrehiew_）。非公式ながら詳細な分解では、Gated DeltaNetハイブリッドパターン（「3層のリニアアテンション：1層のフルアテンション」）を採用し、メモリ使用を抑えつつ品質を維持しているとされています（LiorOnAI）。
  • 実運用への展開が急速に進行：
    • Ollama：ollama run qwen3.5:9b|4b|2b|0.8bでツール呼び出し＋思考＋マルチモーダルをサポート（ollama、ollama）。
    • LM Studio：Qwen3.5-9Bはローカルで約7GBのフットプリントと紹介（Alibaba_Qwen）。
    • iPhoneオンデバイスデモ：「iPhone 17 Pro」でMLXを用い、Qwen3.5 2B 6-bitを動作させた例が「エッジのブレークスルー」として話題に（adrgrondin、kimmonismus）。
  • 評価者向け注意点：小型モデルでは「Reasoning」がデフォルトで無効化されており、chat-templateのkwargsで有効化する必要があります（llama-server / Unsloth docsの例あり）（danielhanchen）。

コーディングエージェント＋信頼性＋「可用性が新たなフロンティア」

  • Codex 5.3とコーディング評価の話題：Codex 5.3が有望なタスクを解決し、WeirdMLベンチマークで79.3%（Opus 4.6の77.9%を上回る）との報告がありますが、Geminiのピーク性能は依然として高い可能性が指摘されています（theo、htihle）。また、WeirdML v2の飽和に近づいているとの推測もあります（teortaxesTex）。
  • 「可用性9の壁に到達しそう」：運用上の課題はモデル品質だけでなくダウンタイムやUX低下にも及び、Claudeの障害による生産性への影響がミームや真剣な苦情として繰り返し共有されています（ThePrimeagen、Yuchenj_UW、Yuchenj_UW）。
  • エージェントの可観測性／評価が一級課題に：
    • 「我々は皆エージェントマネージャーになったが、可観測性を得るお気に入りの方法は？」（_lewtun）。
    • エージェントの信頼性は部門横断的課題（不適切な評価基準はエンジニアリングだけでは解決できず、PMやドメイン専門家が成功定義を担う必要あり）（saen_dev）。
    • 実践的評価アドバイス：構築前に成功条件を定義、まずは決定論的評価者を使用、スタイル評価にはLLMジャッジを活用、プロセスではなく生成物を評価（_philschmid）。
  • AGENTS.md / SKILL.mdは「ガードレール」であり魔法ではない：
    • 10リポジトリ／124 PRでのCodex調査では、AGENTS.mdにより中央値実行時間が約28.6%短縮、トークン使用量が約16.6%削減。これは均一な改善ではなく、主に最悪ケースの無駄動作削減によるもの（omarsar0）。
    • Carnegie Mellon式ループでのSKILL.md改善：「ログ→評価→監視→改善」をOSS例（PRレビューBot）で実装（gneubig）。
  • Anthropic＝コーディング組織化の緊張：「デプロイされたコードの80%以上がClaude Codeによる」との話題が拡散し、速度向上と引き換えに信頼性低下の懸念も（GergelyOrosz）。大企業内でのClaude Code採用や「監督」が手動コーディングを置き換える動きも議論されています（_catwu、Yuchenj_UW）。

インフラ＋ローカルAIハードウェア：Apple Neural Engineの解析、macOSでのDocker/vLLM、そして「AIインフラの年」

  • Apple Neural Engineのトレーニング向けリバースエンジニアリング：研究者が非公開APIを用いてANE上でトランスフォーマートレーニングループを構築、CoreMLを回避。重い処理はANEで、一部の勾配計算はCPUで実行。効率性に関する主張（例：「M4 ANEは6.6 TFLOPS/W、A100は0.08」、「38 TOPSは誇張で実効は19 TFLOPS FP16」）は検証が必要ですが、オンデバイストレーニング／ファインチューニングの可能性を示唆（AmbsdOP、LiorOnAI）。
  • macOSローカル推論の改善：Docker Desktop「Model Runner」がMLXモデルをOpenAI互換APIワークフローで実行可能に。Apple Silicon開発ループの実用的な解放と位置付け（Docker）。
  • 推論ハードウェアの分岐：GPUとTaalas HCの比較では、GPUはHBMストリーミング＋カーネルスケジューリングの制約がある一方、ASICはマスクROMに重みを格納し「モデル＝ハードウェア」として動作。HC1はユーザーあたり16–17k tok/sを達成するが、「1チップ＝1モデル」というトレードオフあり（TheTuringPost）。
  • オープンソース性能解析ツール：AMDがrocprof-trace-decoder（SQTTトレース定義）を公開し、命令レベルのタイミング解析を可能に。AMDのトレーシングインフラは「NVIDIAより優れている」との評価も（tinygrad）。
  • AIインフラが戦略テーマに：Zhipuの「2026年はAIインフラの年」というスローガンは具体性に欠けますが、信頼性＋コスト＋ツールがモデル改善の周縁を凌駕しつつある現状に合致（Zai_org）。
