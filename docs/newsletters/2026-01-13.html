Anthropicのプロダクトスタジオが進化

今回、Cowork と Anthropic Labs の連続発表をまとめてご紹介します。

Coworkは、これまでの Computer Use や Claude CodeをClaude Desktopに統合、Claude for Chrome といった機能を一つのブランドとUI、汎用エージェント「Cowork」に統合したものです。

一方、Labsはよりシンプルな再編で、Mike Krieger氏がCPOを退任し（後任は元Metaの Ami Vora 氏）、現在は彼とBen Mann氏が年間ARR10億ドル超のエージェントラボを率い、Claudeのプロダクト化を進めています。

AI Twitterまとめ

AIエージェント製品：Claude Code/Cowork、LangSmith Agent Builder、そして「エージェント化」された開発ワークフロー

  • Claude Cowork + Claude Codeが「ターミナルネイティブエージェント」の新基準に
    AnthropicのCoworkは Appleのネイティブ仮想化 を使ってLinux VMを起動し、bubblewrap などでサンドボックス化して危険なコマンドや暴走プロセス、誤削除を防ぎます（詳細）。モデルにファイルシステム＋シェル＋厳格な権限を与え、人間レビューを挟みながら高速に反復するUXが標準化しつつあります。一方で、パワーユーザーからは許可プロンプトが多すぎるとの声もあり、--dangerously-skip-permissions を使わずに摩擦を減らしたいという要望が出ています（不満例、ジョーク）。

  • 「Vibe coding」批判とエージェント支援型開発の整理
    「vibe coding」という言葉が、実際にはエージェントと協働しながら検証を行う本格的な開発にも使われてしまっていると議論がありました。Gergely Orosz氏は、コードを検証しループを閉じるエンジニアの作業をvibe codingと呼ぶべきではないと主張（投稿）。Yuchen Jin氏は、vibe codingは本来「コードを全く見ない」ことを意味し、レビューを行えば「lucid coding」に近いと定義（投稿）。これは「エンジニア不要」ではなく、審美眼と検証力を持つエンジニアがより高いレバレッジを得るという変化を示しています。

  • LangSmith Agent Builder GA：ノーコードだが本格派（MCP、メモリ、トリガー、HITL）
    LangChainがAgent BuilderのGAを発表（告知）。メモリ、スキル、サブエージェント、MCP/ツール統合、トリガー、自律実行、そして人間承認用の受信箱などの機能を備えています（解説、まとめ）。技術者にとっても、オーケストレーションと可観測性をパッケージ化している点が有用と評価されています（価値の説明）。

  • 逆トレンド：「モデルの邪魔をせず」ツールを簡素化
    Jared Z氏は、ツールやガードレールの追加が性能低下を招く場合があると指摘し、Vercelがtext-to-SQLエージェントをファイルシステム＋bashアクセスに簡素化した事例を紹介（スレッド）。bash＋ファイルシステムが万能ツールコールであり、現代のモデルは以前必要だった複雑なDAG構造を肩代わりできるという見方です。

  • オープンな再実装と「Coworkクローン」の急速なコモディティ化
    開発者がQEMU＋bubblewrap＋seccompを使ったクロスプラットフォームVMを構築し、vmctlユーティリティとwebsocketで制御（投稿）。MiniMaxはAnthropic互換APIでCoworkを再構築しOSS化した事例を紹介（投稿）。エージェントシェルは再現可能なインフラパターンとなりつつあり、専有的な優位性は薄れています。

長文コンテキスト＋メモリ：RAGチャンク戦争からRecursive Language Models、RLメモリへ

  • ファイルシステム型エージェント vs ベクトル検索：ハイブリッド化が現実的解
    LlamaIndexが「fs-explorer」型エージェントと従来のRAGを比較。ファイル全体のコンテキストを扱えるため精度は高いが遅く、大規模（1,000+文書）ではベクトル検索が優位（投稿、まとめ）。Weaviateはチャンクサイズのトレードオフ（精度 vs 文脈の豊かさ）を再確認（投稿）。

  • MemRL：メモリ検索を類似度検索ではなくRL（効用重視）として扱う
    DAIR AIがMemRLを紹介。ベースLLMは固定し、エピソードメモリに対するQ値を学習。意味フィルタ後に効用ランク付けする二段階検索（概要）。ファインチューニングや忘却を避けつつ、経験から学習するメモリポリシーを構築できます。

  • Recursive Language Models（RLMs）：プロンプトへのシンボリックアクセス
    Omar Khattab氏らは、多くの「サブエージェント」実装が本質を見逃していると指摘。数百万のサブコールをツールコールとして扱うのは非現実的で、プロンプトへのポインタ的アクセスが必要（批判）。The TuringPostは、RLMを推論時アーキテクチャとして説明し、Python REPL変数にコンテキストをオフロードしてコードで操作することで、再学習なしに1,000万トークン超を扱えるとまとめています（概要）。

  • プロンプト圧縮によるコンテキスト劣化対策
    DSPyを使ったプロンプト長削減のワークフローが紹介され、性能を維持しつつコンテキスト劣化を防ぐ方法として提示されています（投稿）。
