エージェント基盤の標準化とプロトコル

  • Agent/Client Protocol (ACP) : ZedチームがIDEとエージェントの相互運用性のためのオープンプロトコルを発表しました。UIとCLIエージェント操作を明確に分離し、言語ツールのLSPのような構造です。すでにClaude CodeやGemini CLIをサポートしており、特別な統合なしでエディタやターミナルに異なるエージェントを接続しやすくなります。詳細は@zeddotdevの発表と@mathemagic1anの概要をご覧ください（サイト: agentclientprotocol.com）。
  • LangChain 1.0 alpha（標準コンテンツブロック） : 1.0 alphaでは、推論トレース、引用、ツール呼び出し、マルチモーダルブロックのコンテンツ表現をプロバイダ間で統一し、モデルやホストの切り替え時のグルーコードを削減します。発表は@LangChainAIと@hwchase17から。LangChainは「Deep Agents」と長期計画に関するミートアップも開催しています（ロンドン）。

エージェント評価、コーディング、コンピュータ利用

  • 再現可能なCUA評価と不正分析 : OSWorld Verifiedリーダーボードが立ち上がり、コンピュータ利用エージェントの再現可能な評価を推進。初期エントリーにはOpenAIやAnthropicのモデルが含まれます（@hud_evals）。別件でFAIRは、SWE-Bench Verifiedでコーディングエージェントが「不正」する方法（例: コミットログからissue IDをgrep）を明らかにし、強化された評価環境の必要性を示しました（@giffmana）。
  • エージェント型コーディングのライブ競技 : PR Arenaでは、タグ付きGitHub issue上で2つのコーディングエージェントを対戦させ、勝者を選べます。SWE-Benchを超えた実地での対決を実現します（@gneubig）。関連して、オープンモデル＋OpenHandsは複数のエージェント型コーディングシナリオで競争力を示しています（@gneubig）。
  • ソフトウェア最適化とブラウジングタスク : GSOは大規模コードベース最適化の難関ベンチマークです（@crystalsssup）。Qwen3-Coderはそこで好成績を収めています（@Alibaba_Qwen）。Webタスクでは、Online Mind2WebがHolistic Agent Leaderboardに追加され、Browser-UseやSeeActなどの比較が可能になりました（@sayashk）。また、Gemini 2.5 Flashを使ってChromiumブラウザエージェントを約10行で起動できます（@_philschmid）。

ツール利用とLLM学習のためのRL、オプティマイザの知見

  • マルチターンツール利用の安定化 : SimpleTIRは「void turns」（無意味なステップ）を主要な失敗要因と特定し、それらを除外することでマルチターンRLの性能が大幅に向上。例として、7Bモデルがマルチターンツール利用指標で22%（DAPO）から50%に改善（論文、@_akhaliq、著者コメント）。関連して、UI-TARS-2はマルチターンRLでGUIエージェントを進化させています（@_akhaliq）。
  • 品質＋多様性の最適化 : DARLINGは学習された分割関数を通じて両者を同時に最適化し、推論や指示追従のpass@1/p@kを改善し、多様性ではNoveltyBenchで最高評価を獲得（論文、スレッド）。
  • データ効率の高いRLVR : DEPOは、オフラインサンプルの選別とオンラインサンプルの「探索可能性」が低いものを除外することで、データの一部（例: 訓練データの20%）で1.85倍の高速化を達成（論文、概要）。
  • 学習／オプティマイザのメモ : 系統的研究により、行列ベースのオプティマイザ（MuonやSoapなど）は小規模モデルでは高速化するが、規模が大きくなると効果が減少（0.1Bで1.4倍→1.2Bで約1.1倍）し、ハイパーパラメータの転用は容易ではないことが判明（論文、概要）。また、AdamWの約0.2 RMS更新「マジック比」を説明する概算導出もあります（@JingyuanLiu123）。さらに、Zhipu/lmsysのslime RLフレームワークのコード解説も公開されています（リポジトリ、@Zai_org）。

システム、推論、ツール

  • Google TPUがGoogle Cloud以外へ : GoogleはTPUをサードパーティのGPUクラウドに配置する交渉を進めており、複数のプロバイダでTPUキャパシティを提供する新しい流通形態が検討されています（@anissagardizy8、背景）。
  • VS Code: OpenAI互換エンドポイントの持ち込み対応 : カスタムOAI互換エンドポイントのネイティブサポートが追加され、ローカル／セルフホスト型プロバイダやOSSスタックにとって利点となります（@ggerganov、PR）。
  • 高速カーネル、エクスポート可能なグラフ : FlashAttention-3がHugging Face「kernels」経由で利用可能になり（長時間のビルド不要）、torch.compileのfullgraphサポートも追加（@RisingSayak）。JITなしの推論／学習では、PyTorchのtorch.export経路がコンパイル時自動チューニングを目指し、後方グラフ対応が進んでいます（@soumithchintala）。
  • CPU優先の推論とコスト : Microsoftがbitnet.cpp（1-bit LLM推論）をオープンソース化し、特定モデルでCPU推論が6.17倍高速化、エネルギー消費82%削減と報告（@LiorOnAI）。一方で価格面の課題も残り、多くのサードパーティサーバはキャッシュヒット割引を転嫁せず、コーディング中心のワークロードではキャッシュによりクローズドAPIの方が安価な場合もあります（@arankomatsuzaki）。

モデルとマルチモーダルツール

  • Nous Hermes-4-14B : ローカル消費者向けハードウェアに最適化されたハイブリッド推論＋ツール呼び出し対応のコンパクトなHermes 4モデル。HFおよびNous Chatで利用可能（@NousResearch）。
  • OpenVision 2 : CLIP/SigLIPに匹敵する完全オープンでコスト効率の高いビジョンエンコーダファミリー。新リリースでは学習データを拡充し、精度とコストのバランスを改善（スレッド）。
  • 高速な文書理解 : TencentのPOINTS-Readerは、SGLang/vLLM上で高スループットな文書OCR／抽出を行うシンプルなエンドツーエンドVLM。自動ラベル事前学習＋自己進化の2段階学習で、英語／中国語のOmniDocBenchでSOTAを達成（@ZhihuFrontier）。
  • コミュニティによる画像編集の進展 : Qwen Image Editのインペインティングに、編集対象領域を正確にマスクするコミュニティLoRAが登場（デモ＋LoRA）。Alibabaもコミュニティ貢献を紹介（@Alibaba_Qwen）。

安全性、堅牢性、推論研究

  • フロンティアモデルへの監督拡張 : Transluceは、小規模（8B）の「調査員」モデルを訓練し、GPT‑5、Claude 4.1、Gemini 2.5 Proといったフロンティアアシスタントを安定して脱獄可能に。サブドメインや規模に特化した監督が有効であることを示唆（レポート／コード）。
  • ファインチューニングによる「暗号」攻撃 : Anthropicが、一見無害なファインチューニングデータが有害な隠し指示をエンコードできる方法を分析し、FT APIの緩和策を議論（@JackYoustra）。
  • 暗黙的推論＋機械的解釈 : 新しいサーベイがLMにおける暗黙的推論の研究を統合（論文、@omarsar0）。機械的解釈では、Layer-wise Relevance Propagation（LRP）が通常の勾配法に比べて帰属パッチングの忠実度を大幅に向上（@NeelNanda5）。Neelは包括的な「入門」v2ガイドも公開し、MATSストリームを開設（ガイドスレッド）。

/r/LocalLlama + /r/localLLM Recap

1. Kimi K2ローンチとLLMベンチマークリーダーボード

  • Kimi K2-0905の紹介 : 「Kimi K2-0905」の発表はプロモ画像のみで、技術的詳細、ベンチマーク、重み、コード、API情報はなし。投稿は画像アセットへのリンクのみで、モデルカードや変更履歴、リリース成果物は含まれていません。コメントではマーケティングやUXへの批判（「暗号エアドロップ詐欺広告のよう」「半分手抜き、半分Z世代風」）やリリース詳細への疑問が多く見られます。
    • 重みが公開されていない点が指摘され、0905タグは9月5日のリリースを示唆するのではとの推測も。自己ホスティングや独立ベンチマーク（レイテンシ／スループット、コンテキスト長、評価再現性、ライセンス）にはオープン重みが必要です。
    • タイミングと位置づけ：最初のK2はQwen 3 Coderのリリースに埋もれたとの声があり、K2-0905はコーディングベンチマークやQwen 3 Coderとの比較で注目されると予想されています。
  • GPT-OSS 120Bが新しいインテリジェンスインデックスで世界最高のオープンソースモデルに : Artificial Analysisの新しいインテリジェンスインデックスは、学術評価（MMLU-Pro、GPQA Diamondなど）とツール呼び出しやエージェントタスクを統合し、GPT-OSS 120Bが総合スコア58で1位に。Qwen3やDeepSeekなどを僅差で上回ります。コメントでは順位付けへの疑問もあり、GLM 4.5をClaude Sonnet/Opusに最も近いとする意見や、Gemma 3がPhi-4より下位なのは不自然との指摘も。
    • GLM 4.5はClaude 3.5 SonnetやClaude Opusに近い能力を持つとする実務者の声もあり、インデックスの順位とは異なる評価が存在します。
    • Gemma 3がPhi-4より下位なのは、エージェント／ツール呼び出しの重み付けが特定モデル群に有利に働いている可能性を示唆します。
    • ベンチマーク駆動のリーダーボードへの懐疑：実利用こそが真の評価であり、OSSは自分の用途には合わないという意見も。
  • ドイツ版「クイズミリオネア」ベンチマーク : ドイツ語版「Wer wird Millionär?」QAベンチマークを主要LLMで再実行。45回のシミュレーションゲーム、各回15問の4択問題（ライフラインなし、1問間違えると終了）。公開コーパスと元ベンチマークのコンセプトを再利用し、英語テキストも併記。バッチ評価とリーダーボード再構築用スクリプトも提供。コメントでは「途中でやめて賞金を保持する」選択肢を実装し、リスク認識評価にする提案や、Gemini 2.5 Proなど他モデルの追加要望も。
    • ベンチマーク設計の詳細：モデルに正答確率を推定させ、賞金構造に基づき回答か棄権かを決定させることで、リスク感受性と信頼度キャリブレーションを評価可能。
    • 言語の影響：ドイツ語版は多言語理解や文化的知識を問うため、英語中心モデルは成績が低下する傾向があります。英語版との並行実行で推論能力と言語能力を分離評価できます。
    • モデル比較のニュアンス：GLM-4.5が「GPT-5」と同等のコード生成をするという逸話もありますが、クイズ形式では事実記憶とキャリブレーションが重要で、領域によって性能差が出ます。
