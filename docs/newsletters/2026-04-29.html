コーディングエージェントがプラットフォーム化：Codex、Cursor SDK、VS Code Harnessアップグレード

  • OpenAIはCodexをコーディングツールから汎用作業環境へと進化：今日最も強い製品シグナルは、単なる利用熱ではなく、永続的コンテキスト、ツール、統合、チーム展開といった機能拡張の着実な進展でした。OpenAIはCodexをコードに加え、研究の統合、スプレッドシート、意思決定の記録など幅広い知識業務に活用できると強調しました（OpenAI、続報、続報）。さらに、6月末までの期間限定で、対象のBusiness/Enterprise顧客向けにCodex専用席を席料$0で提供（OpenAIDevs）、Supabaseとの統合（coreyching）、実装計画をFigJamボードに変換するFigmaプラグイン（OpenAIDevs）などを追加しました。コミュニティ投稿では、アプリサーバー利用やより豊かなエージェントワークフローも指摘されています（gdb、aiDotEngineer）。
  • 性能改善はモデルのレイテンシからエージェントループのシステム工学へ移行：OpenAIはCodex型ワークフローをResponses APIのWebSocketモードに移すことで、ツール呼び出し間で状態を保持し、繰り返し作業を削減し、最大40%高速化できると説明しました（OpenAIDevs、reach_vb、pierceboggan）。VS Codeも並行してハーネス改善を実施：ワークスペース全体のセマンティックインデックス、クロスリポジトリ検索、チャットセッションインサイト、スキルコンテキスト、Copilot CLIのリモート制御、プロンプトやスキル、指示を改善する評価拡張機能など（pierceboggan、pierceboggan、code）。全体として、コーディングエージェントのUXは、単なるモデル知能ではなく、メモリ、検索、ハーネス品質、ツールのオーケストレーションが支配的になっています。
  • Cursorは明確なプラットフォーム戦略を展開：新しいCursor SDKは、Cursorを支えるランタイム、ハーネス、モデルをCI/CD、オートメーション、製品内の埋め込みエージェントに利用できるよう公開しました（cursor_ai、スタータープロジェクト、顧客事例）。これは、Cursorを席ベースのIDE製品からプログラム可能なエージェントインフラへと転換するものであり、この方向性は@kimmonismusによって的確に表現されています。CodexのアプリサーバーやVS Codeのハーネス改善と合わせると、カテゴリは明らかにヘッドレスエージェントランタイム＋プログラム可能なハーネス＋利用量ベースの経済へ収束しています。
