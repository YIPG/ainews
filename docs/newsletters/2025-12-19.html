オープンなマルチモーダル＋「クリエイティブツール」リリース（Qwen Image Layered、Kling Motion Control、Runway GWM）

  • Qwen-Image-Layered（ネイティブ画像分解、オープンソース）：AlibabaがQwen-Image-Layeredを公開しました。「Photoshop級」のレイヤー画像分解機能を持ち、物理的に分離されたRGBAレイヤーを出力します。プロンプトで構造を制御でき、3〜10レイヤーを指定可能。さらにレイヤー内にレイヤーを持つ「無限分解」も可能です。発表スレッドにはHF/ModelScope/GitHubや技術レポートへのリンクがあります（@Alibaba_Qwen）。コミュニティでは編集のしやすさやテキスト分離品質が評価されています（@linoy_tsaban、@linoy_tsaban）。falなどの提供プラットフォームにも早速登場しています（@fal）。
  • Kling 2.6 Motion Control（画像から動画への制御＋クリエイター向けループ）：複数の高反応デモで、モーションコントロールがプロンプトのみの制御を超えたキャラクターアニメーションの実用的手段として示されています。特にv2vワークフローで有効です（@onofumi_AI、@blizaine）。KlingはMotion Controlに関する公式コンテストも開始しました（@Kling_ai）。クリエイターは高アクション動作のプロンプト「レシピ」も共有しています（@Artedeingenio、@StevieMac03）。
  • RunwayのGWM-1ファミリー＋Gen‑4.5アップデート：RunwayはGWM Worlds / Robotics / Avatarsを発表しました。これはカメラ動作の一貫性と応答性を持つフレームごとの動画生成を特徴とします。Gen‑4.5では音声やマルチショット編集が追加されました。概要は@DeepLearningAIから、@c_valenzuelabによる「シーケンスショット」への期待も寄せられています。

LLMプラットフォームの動き：Gemini 3 Flash vs GPT‑5.2、「RLがFlashをProより優位に」、ベンチマークの変動

  • Gemini 3 Flashの勢い（ツール＋エージェント的UX）：複数のベンチマークでGemini 3 Flashがツール利用や総合指標でトップとされています。例：「Toolathlonで#1」（tweet）、EpochAIのECIでGPT‑5.2を上回る（tweet）、SimpleBenchでGPT‑5.2 Proより上位の5位（tweet）。Googleの製品展開では音声からプロトタイプや広範な統合が強調されています（@Google、@GeminiApp）。
  • 「FlashがProを超えた理由はRL」：FlashがProより優れているのは単なる蒸留ではなく、新しいエージェント的RLを導入したためという主張があります（@ankesh_anand）。これは「Flashが魔法」というより、リリース時期＋後処理レシピが「ファミリー内の序列」を左右するという教訓です。
  • GPT‑5.2の使用メモ（長文コンテキスト＋ツール）：一部ユーザーは約256kトークンまでの長文タスクでGeminiよりGPT‑5.2を好むと報告しています（@Hangsiin）。ただしChatGPTのUX（ファイルアップロード＋検索動作）が「完全なコンテキスト統合」を阻む場合があり、パワーユーザーはCodex CLIへ移行しています（@Hangsiin）。
  • モデルの信頼性／「劣化」議論（Anthropic Opus 4.5）：複数の投稿でOpus 4.5の劣化／ループ化が指摘され（tweet、@Teknium）、ユーザーが「心を読んでくれる」ことを期待するワークフロー状態に入ることも「劣化」と感じる要因だとの見方もあります（@kylebrussell）。

エージェントエンジニアリングを製品化：ハーネス、評価基盤、Codex「Skills」、可観測性

  • 「エージェントとハーネスは完全に結合」：共感を呼んでいる具体的なメンタルモデル：

    • エージェント＝モデル＋プロンプト＋ツール／Skills／MCP＋サブエージェント＋メモリ

    • ハーネス＝実行ループ＋コンテキスト管理＋権限／リソースポリシー

    重要なポイント：ハーネスは製品として提供され、サブエージェント／ツール／プロンプト＋UX機能（計画モード、圧縮ポリシー、切り捨て／オフロード）を含みます（@Vtrivedy10、tweet）。 * Codexが「Skills」を追加（エージェント機能の標準化）：OpenAIはCodex Skillsを導入しました。これは指示／スクリプト／リソースの再利用可能なバンドルで、$.skill-nameで呼び出すか自動選択されます（@OpenAIDevs）。例としてLinearチケットの読み書きやGitHub CI失敗の自動修正があります（Linear skill、CI skill）。@gdbはagentskills.ioの「標準」との整合性を指摘し、相互運用可能なエージェント機能モジュールへの動きを示唆しています。 * コーディングエージェントのトレーシング＋安全性： * Claude Code → LangSmithトレーシング：LangChainはClaude Codeの全LLM／ツール呼び出しを観測する統合機能を提供しました（@LangChainAI、@hwchase17）。 * AgentFSがCodex対応を追加：LlamaIndexはファイルシステムのサンドボックス機能をCodexやOpenAI互換プロバイダーに対応させました（@llama_index）。 * 評価基盤がボトルネックに：Jared Palmerは大規模リファクタリングでは「半分の時間」がハーネスエンジニアリング（Skills、エージェント、コマンド、テスト設定、検証）に費やされると述べ、ゲームエンジン開発に似ていると指摘しています（@jaredpalmer）。補足として、エージェントが生成するコード量が増えることでレビューがボトルネックになるとの見方もあります（@amanrsanger）。
