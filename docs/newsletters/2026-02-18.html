AI Twitterまとめ

フロンティアモデルとベンチマーク更新（Claude 4.6、Qwen3.5、GLM‑5、Gemini 3.1 Pro、MiniMax M2.5）

  • Anthropic Claude Opus/Sonnet 4.6：大幅な性能向上と高額なトークンコスト
    Artificial Analysisによると、Sonnet 4.6はIntelligence Indexで51（Sonnet 4.5の43から上昇）となり、Opus 4.6の53に迫る性能を示しました。ただしトークン効率は悪化し、スイート実行に必要な出力トークンは約74M（Sonnet 4.5は約25M、Opus 4.6は約58M）で、最大実行時のコストは$2,088に達します。
    コミュニティでは「4.6は批評やアーキテクチャ設計が得意になった」との声がある一方、Claude Codeの信頼性や製品面での課題も指摘されています（SDKやドキュメント、ツール安定性に関する議論）。
  • ClaudeのSearch Arena追加と自律性計測
    Arenaは検索モードのリーダーボードにOpus/Sonnet 4.6を追加。Anthropicは「実際のAIエージェント自律性の測定」を発表し、数百万件のツール使用インタラクションを分析。ツール呼び出しの約73%は人間が介在し、0.8%のみが不可逆的、ソフトウェアエンジニアリングがAPI上のツール呼び出しの約50%を占めると報告しました。
  • Qwen 3.5：推論効率と“過剰思考”問題
    Qwen3.5の「考えすぎ」やトークン消費が議論され、Qwen3.5-Plusでは長い推論チェーンのトークン膨張が減少した一方、非推論モードでは性能低下が見られるとの分析も。Qwen3.5-PlusはVercel AI Gatewayに提供され、Alibaba Cloudは高リクエスト上限の固定料金制「Qwen Coding Plan」を開始。
  • Qwen3.5-397B-A17B FP8ウェイト公開
    AlibabaはQwen3.5‑397B‑A17BのFP8ウェイトを公開し、SGLang対応を統合、vLLM対応も近日予定。オープンウェイトと即時エコシステム対応が競争力の標準となりつつあります。
  • GLM‑5技術報告と“エージェント型エンジニアリング”RL基盤
    GLM‑5は生成と学習を分離する非同期エージェントRLや、長文コンテキスト性能を維持しつつ計算負荷を減らすDSAを導入。OSS再現に有用な詳細な技術報告として評価されています。
  • Gemini 3.1 Proの噂と“長く考える”傾向
    初期テストではGemini 3.1 ProがGemini 3 Proより長い推論を行い、OpusやGPTに近づく可能性が示唆される一方、ベンチマーク信頼性や敵対的ケースでの失敗も指摘されています。
  • MiniMax M2.5がコミュニティリーダーボードに登場
    MiniMax M2.5とM2.5 Lightningがオンボードされ、プロンプト投票型リーダーボードで結果が追跡されています。

エージェント型コーディングとハーネス設計（Claude Code、Cursor、LangSmith、Deep Agents、SWE-bench）

  • ハーネスが性能を左右
    同一モデル（Claude Opus 4.6）でも、LangChain Deep Agents CLIは9秒で完了、Claude Codeは16秒と1.7倍の差。オーケストレーションやツールポリシー、コンテキスト戦略がユーザー体感性能を支配することが示されています。
  • Cursorが“エージェントメモリ”UXを強化
    .agents/skills対応に続き、過去の会話をコンテキストとして利用可能に。IDEエージェントの永続的かつツール利用可能な記憶に向けた実用的な一歩です。
  • LangSmith Agent Builderのアップグレード
    全ワークスペースツールにアクセスできる「汎用エージェント」チャット、チャット→エージェント変換、ファイルアップロード、中央ツールレジストリを追加。実験からデプロイまでの摩擦を減らす狙いです。
  • SWE-benchインフラ更新
    リーダーボードがmini-SWE-agent v2で全実行に移行し、ベースモデルの性能解釈が変化。評価方法への不満も続いています。
  • Windowsエージェントシェルの安全上の落とし穴
    Git Bash/MSYS2で2>nulのようなWindowsリダイレクトを出すとNTFS上に削除できないnulファイルが生成されるため注意。Unixスタイルのリダイレクトかcmd /cでラップする必要があります。

OpenAIとスマートコントラクトセキュリティ評価（EVMbench）

  • EVMbench開始
    OpenAIは高危険度スマートコントラクトの脆弱性検出・悪用・修正能力を評価するEVMbenchを導入。エージェント型セキュリティが主要評価カテゴリとなりつつあります。
  • エンジニア向けのシグナル
    実際の悪用・修正ワークフローに近い評価であり、エージェント型コードレビューやオンチェーン監視、インシデント対応の構築に有用です。

データ、キュレーション、評価の衛生管理

  • ÜberWeb：英語性能を犠牲にせず多言語性能向上
    DatologyAIの「ÜberWeb」は20兆トークンスケールで多言語モデルの性能向上を実現し、「多言語の呪い」は主にデータ品質の問題だと反論。
  • プロンプト繰り返し論争
    同じプロンプトを2回繰り返すと精度が大幅に向上するという主張に対し、質問を最初に置くと効果が消える可能性や、ベースライン欠如による結果の誇張が指摘されています。
  • データセット汚染はもはや仮説ではない
    数十年残存した誤った「最初の500個の素数」ページが2026年の生成AIモデルを汚染する事例が共有され、事実の脆弱性と出典管理の必要性が浮き彫りになっています。
  • AIスロップ検出と出典確認
    実在しないロボット製品の偽メディアなどが警告され、GoogleはGemini内の音声にSynthID透かし検証を導入し、画像・動画に加えて音声の出典確認を拡張しています。

マルチモーダルとクリエイティブモデルのリリース

  • Google/DeepMind Lyria 3：Geminiに音楽生成機能を搭載
    Lyria 3はテキストや画像/動画プロンプトから30秒の曲を生成し、歌詞やボーカルも対応。SynthID透かしを付与し、Geminiで音声出典確認が可能。ジャンル・雰囲気・楽器・ボーカル・歌詞を構造的に指定することで制御性が向上します。
  • OriOn：長文コンテキスト対応VLMによる文書検索
    LightOnは最大250ページの高解像度視覚情報を一度に処理できる長文コンテキストVLM「OriOn」を発表し、学習レシピと修正版ベンチマークMMLBD‑Cを公開。
  • 動画生成・編集研究が続々
    空間記憶による一貫性生成やリアルタイム編集の分離制御など、時間的一貫性におけるリトリーバルと構造化メモリが重要テーマとして浮上しています。
