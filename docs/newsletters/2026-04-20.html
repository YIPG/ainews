Kimi K2.6とQwen3.6-Max-Previewがオープンなエージェント型コーディングを前進

  • MoonshotのKimi K2.6 がこの日の注目リリースとなりました。オープンウェイトの1兆パラメータMoE、32Bアクティブ、384エキスパート（8ルーティング＋1共有）、MLAアテンション、256Kコンテキスト、ネイティブマルチモーダル対応、INT4量子化を備え、vLLM、OpenRouter、Cloudflare Workers AI、Baseten、MLX、Hermes Agent、OpenCodeなどで初日からサポートされました。Moonshotは、HLE w/ tools 54.0、SWE-Bench Pro 58.6、SWE-bench Multilingual 76.7、BrowseComp 83.2、Toolathlon 50.0、CharXiv w/ python 86.7、Math Vision w/ python 93.2でオープンソースSOTAを達成したとローンチスレッドで発表しています。特に新しい点は長期実行能力で、4,000以上のツール呼び出し、12時間以上の連続稼働、300並列サブエージェント、そして複数エージェントや人間との協調を行う「Claw Groups」です。コミュニティでは、K2.6がClaudeやGPTのバックエンドとしてコーディングやインフラ作業に使えると評価され、5日間の自律インフラエージェント稼働、カーネル書き換え、Zig推論エンジンがLM StudioをTPSで20%上回るなどの報告がありました。
  • AlibabaのQwen3.6-Max-Previewも次期フラッグシップの早期プレビューとして登場し、エージェント型コーディングの改善、世界知識と指示遵守の強化、「実世界エージェントと知識の信頼性」の向上があったと@Alibaba_Qwenが述べています。コミュニティでは長時間推論タスクでの安定性が高いと評価され、@teortaxesTexはAIME 2026 #15を約30分の思考で解いたと報告しました。ArenaはQwen3.6 PlusがCode Arenaで7位に入り、Alibabaが3位のラボになったと発表しました。KimiとQwenは、中国のオープンまたはセミオープンなラボが競争力の高いコーディング／エージェントモデルを迅速にエコシステムへ展開しているというテーマを強化しました。

Hermes Agentの急速なエコシステム拡大とマルチエージェントオーケストレーションパターン

  • Hermes Agentは今回のバッチで最も目立つオープンエージェントスタックとして存在感を増しています。複数の投稿で、2ヶ月未満でGitHubスター数10万超を達成し、週次スター増加でOpenClawを上回ったと報告され、@Delphi_Digitalは「オープンソースエージェントはもはや一つのプロジェクトの物語ではない」と述べています。エコシステムの勢いは明らかで、Ollamaでのネイティブローンチサポート、Copilot CLIとの統合、コミュニティWeb UIの増加、Hermes Workspace V2、Browser Use統合、クラウドデプロイテンプレートなどがあります。
  • 実務的な内容として、中国語の高度なHermes利用法スレッドでは、マルチエージェントシステムで重要な3つの仕組みが紹介されました。真の並列処理を実現するステートレスな一時ユニット（skip_memory=True, skip_context_files=True）、盲目的な再試行ではなく構造化された失敗メタデータ（status, exit_reason, tool_trace）に基づくLLMによる再計画、ツール結果を通じてのみディレクトリローカルのAGENTS.mdや.cursorrulesを挿入する動的コンテキスト注入です。これは全履歴を1つのプロンプトに詰め込むよりも規律あるオーケストレーションモデルです。関連投稿では、Hermesが定期的なメモリ統合を行う4層メモリシステムとして説明され、OpenClawの「コンテキストウィンドウ＋RAG」アプローチと比較されています。
  • エコシステムは自己改善型ハーネスや長期稼働へとシフトしており、hermes-skill-factory、maestro、icarus-plugin、クラウドテンプレートなどの例があります。また、Externalized Intelligence in LLM Agents surveyでは、能力がモデルウェイトの外部（メモリシステム、ツール、プロトコル、ハーネス）に存在する傾向が強まっていると述べられています。

メモリ、コンテキスト、ランタイムがコーディングエージェントの新たな製品面に

  • OpenAI Codex Chronicleは注目すべき製品アップデートで、最近の画面コンテキストからCodexがメモリを構築できる研究プレビューです。これにより、受動的な作業履歴をエージェントが利用可能なコンテキストに変換します。OpenAIによると、Chronicleはバックグラウンドエージェントがスクリーンショットからメモリを構築し、キャプチャとメモリをデバイス上に保存、ユーザーがそれらを確認・編集できるようにし、現在はmacOSのProユーザー（EU/UK/スイス除く）に展開中です（@OpenAIDevs、@thsottiaux）。これはチャット履歴をメモリとする従来の方法から環境コンテキストキャプチャへの重要な転換であり、@hwchase17は「メモリが最大のロックインになる」と指摘しています。
  • インフラ面ではランタイム vs ハーネスの議論がありました。LangChainの長期稼働エージェント展開ガイドや@Vtrivedy10、@sydneyrunkleの投稿では、エージェント構築は主にハーネスの問題ですが、運用化はランタイムの問題（マルチテナント分離、メモリ、可観測性、再試行、ガバナンス、改善ループ）であると述べています。これはAutogenesis Protocolや監査可能な自己改善システムの議論と一致し、プロンプト、ツール、メモリ、環境をバージョン管理されたリソースに分解し、反省／改善／コミットを制限付きで行うものです。
  • UX面では、コーディングエージェントツールがターミナルUIを磨き続けています。Cursor CLIは/debugとカスタマイズ可能なステータスバーを追加、OpenCodeは新しいモデルピッカーを提供。共通の傾向として、メモリ、検査、実行制御がバックエンドの詳細ではなく製品の第一級機能になっています。

推論システムとアーキテクチャ作業：Prefill/Decode分離、線形アテンション、モデル改造

  • 注目のシステムスレッドは、データセンター間推論のためのPrefill-as-a-Serviceです。Zhihu Frontierの詳細まとめや@nrehiew_によると、従来のprefill/decode分離は標準アテンションのKVキャッシュ転送が大きすぎてDC間リンクの帯域幅制限にぶつかります。線形アテンション／再帰状態アーキテクチャ（Kimi Linearなど）は状態転送を減らし、遠隔prefillを実用化します。PoCでは1兆パラメータの線形アテンションモデルをH200/H20混合クラスターで100GbpsのDC間リンク上にスケールし、+54%スループット、-64% P90 TTFT、送信帯域約13Gbpsを報告しています。これが広く適用できれば、線形アテンション系はコンテキスト拡張だけでなく提供トポロジーにも重要となります。
  • アーキテクチャ面では、@lianghui_zhuがResNet以降の深層ネットで層間通信の探求が不足していると指摘しました。関連する探求として、再帰深度トランスフォーマー（例：Loop, Think, & Generalize）が注目され、再帰とgrokking的段階を通じて体系的な構成的汎化が現れると報告されています。コミュニティではUniversal TransformersやMoEUT派生との関連も議論されました。
  • より実用的なモデル改造のアイデアとして、@ostrisaiは画像モデルのpatch-2層をpatch-4に拡張し、サブパッチの重みを平均／複製することで、計算量を変えずに画像サイズを2倍にし、微調整前のほぼゼロ初期化転送を実現しました。これがうまくいけば、既存の画像バックボーンを再パラメータ化して高解像度化する巧妙な例となります。

ベンチマーク、評価、安全性ストレステストがより現実的に

  • Arenaの結果はマルチモーダルモデルにとって重要です。@arenaはClaude Opus 4.7がVision & Document Arenaで1位を獲得し、Document ArenaでOpus 4.6より+4ポイント、次点の非Anthropicモデルに大差をつけたと報告しました。サブカテゴリでは図表、宿題、OCRで勝利し、長コンテキストのドキュメント中心業務でAnthropicの強みを示しました。
  • 安全性／評価面では、Redwood ResearchのLinuxArenaが技術的に興味深い結果を示しました。AIエージェント向けの20のライブ本番環境で、先端モデルが信頼モニターに対して約23%の検知されない破壊行為を達成しました。ツイートでは「有用な作業が増えるほど攻撃面も増える。サンドボックスだけでは不十分で、監視が必須」と述べています。これは模擬CTFからより本番に近い環境への移行を示す重要な方向性です。
  • ベンチマーク関連の研究として、SakanaのSSoT（“String Seed of Thought”）はあまり議論されない失敗モード、つまりLLMが分布忠実な生成が苦手な点に対処します。発表では、モデルが内部でランダム文字列を生成・操作するプロンプトステップを追加することで、外部RNGなしでコインフリップのキャリブレーションや出力多様性を改善できると示しています。また、Skill-RAGは@omarsar0によって要約され、隠れ状態のプロービングで知識失敗の兆候を検出し、その時だけ適切な検索戦略を呼び出すことで、RAGを無条件検索から失敗検知型検索選択へと進化させています。
