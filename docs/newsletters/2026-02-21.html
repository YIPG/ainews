AI Twitter Recap

フロンティアモデル評価：Gemini 3.1 Pro、SWE-bench、MRCR、そして「二極的」な実世界性能

  • Gemini 3.1 Proは強力な検索性能＋エージェント的利用性は混在 : Context ArenaのMRCRアップデートによると、Gemini 3.1 Pro Previewは簡易検索（2-needle @128k AUC 99.6% vs 99.8%）ではGPT‑5.2 (thinking:xhigh)とほぼ同等、難易度の高い複数検索（8-needle @128k AUC 87.8%）ではGPT‑5.2のthinking tierを上回るとのことです（DillonUzar）。別の報告ではArtificial Analysisがトークン効率＋価格の重要性を指摘し、同社のIntelligence Indexスイートの実行コストはGemini 3.1 Pro Previewで$892、GPT‑5.2 xhighでは$2,304、Opus 4.6 maxでは$2,486であり、消費トークン数もGPT‑5.2より少なかったとしています（ArtificialAnlys）。
  • しかしエンジニアからは「ベンチ強いが製品弱い」との声 : Geminiのツールやハーネスが遅れているとの複数スレッドがあり、CLIでのモデル利用可否の不安定さや「Antigravity」でのバグ、UIとモデル表示の不一致（アプリはGeminiと表示しつつ内部ではClaude）などが報告されています（Yuchenj_UW、Yuchenj_UW）。「速い馬」と評価する肯定的意見もある一方、日常利用での不満も目立ちます（theo）。
  • SWE-bench Verifiedの評価手法が再び重要に : MiniMaxは、同一設定下でのMiniMax M2.5のSWE-bench Verified結果を独自に検証し、以前のラボ間比較が不適切だった可能性を示唆しています（MiniMax_AI）。Epoch AIもこの問題を認め、評価手法を更新した結果、開発者報告値に近い結果が得られるようになったとしています（EpochAIResearch）。
  • ベンチマークの奇妙さが「何を測っているのか」議論を促す : 例として、フロンティアモデルは「ARC-AGI」を圧倒する一方でConnect 4では苦戦しており、ARC型パズルは空間・ゲーム推論の一部しか捉えていない可能性が指摘されています（paul_cal）。別スレッドではARC‑AGI‑3用の「シンプルハーネス」で進展できるモデルは少数で、コストが制約になると予想されています（scaling01、scaling01）。

Claude Opus/Sonnet 4.6：タイムホライゾン評価、コスト、信頼性の課題

  • METRによるOpus 4.6の「タイムホライゾン」大幅増加、ただし推定はノイズ大 : METRはClaude Opus 4.6のソフトウェアタスクにおける50%タイムホライゾンを約14.5時間（信頼区間 6–98h）と報告し、タスクスイートが飽和状態に近く測定が「極めてノイズが大きい」と警告しています（METR_Evals）。METRスタッフはタスク分布のわずかな変化が測定値に大きく影響する可能性を再確認しています（idavidrein）。外部コメントでは、ステップごとのエラー率が非常に低い場合、小さな改善が全体成功率に大きく影響するとの指摘があります（xlr8harder）。
  • トークン制限＋長時間推論が実務上の失敗要因 : Opus/Sonnetが最大トークン制限に達し、長時間の「思考」後に空出力となる事例が複数報告されており、「最大推論」がUXやコスト面でのリスクになっています（paul_cal、htihle）。
  • Arenaのシグナル：Sonnet 4.6がCode Arenaで急上昇 : ArenaによるとSonnet 4.6はCode Arena WebDevで#3に上昇（4.5では#22）し、命令遵守や数学カテゴリでも改善が見られます（arena、arena）。
  • Claude Code製品の不安定さが反発を招く : Claude CodeのUXや性能低下（タイムスタンプ、思考表示欠如、長時間ハング）や「ゼロからの書き直し」感が議論を支配しました（theo、theo）。これと同時にOpenCodeへの法的圧力（Anthropicの弁護士からの書簡とされる）が話題となりました（theo）。

エージェント、スキル、オーケストレーション：GEPA/gskill、RLM、そして「エージェントスタック」の形式化

  • GEPA for Skills / gskill：プロンプト＋スキル最適化がパイプライン化 : 複数の投稿でgskillが紹介され、GEPAを用いてエージェントの「スキル」を自動学習するパイプラインとして、リポジトリタスク解決率ほぼ100%、Claude Codeで学習スキル使用時に47%高速化を達成したと報告されています（ShangyinT）。ワークフローはSwe‑Smithでリポジトリタスク生成→GEPA optimize_anythingでスキル最適化→スキルファイル納品という流れです（AlexGDimakis）。DSPy Weeklyもこれを重要なエコシステムの一歩と位置付けています（getpy）。
  • スキルは新たな「ソフトウェア成果物」かつ新たな故障面 : スキルは最小限で人間が慎重に書くべきか、モデル生成の膨大なドキュメントにすべきか議論があり、「少ない方が良い」派は2段落の凝縮ガイドが20ページの自動要約より優れると主張しています（hrishioa）。一方、「スキルダウンタイム」のような運用事故は、スキルがネットワーク依存になると他サービス同様の信頼性問題を引き継ぐことを示しています（theo）。
  • RLM（Recursive Language Models）がメタハーネスとして登場 : 複数投稿でRLMが多様なハーネスを「自発的に」模倣できる汎用ワークフロー基盤として扱われています（HammadTime）。OmarはGPT‑5.2‑CodexやGemini 3.1 ProがRLM分解戦略と相性良く動作する一方、Opus 4.6はこのパターンでは劣ると報告しています（omarsar0、omarsar0）。
  • オーケストレーションが差別化要因に : 論文要約によると、モデルのベンチ性能が収束する中で、マルチエージェントオーケストレーショントポロジー（並列／順次／階層／ハイブリッド）が最適化対象として重要になり、トポロジールーティングで12–23%の性能向上が得られたと報告しています（omarsar0）。Anthropicの利用テレメトリーでは監督は「全ステップ承認」ではなく「必要時介入」が主で、興味深いことにエージェントは人間の介入よりも頻繁に確認を求めるとされています（omarsar0）。

ローカル／オープンツール＋インフラ変化：ggml/llama.cppがHugging Faceに参加、Ollama統合、推論経済

  • オープンソースの大規模統合：ggml.ai（llama.cpp）がHugging Faceに参加 : Georgi Gerganovはggml.aiがHFに参加し「ローカルAIを簡単かつ効率的にする」と発表しました（ggerganov、huggingface）。コミュニティでは、llama.cppが2023年初頭に始めた「ローカルモデル革命」が制度化されたと捉えています（simonw、victormustar）。
  • ローカルファーストはトークン不足経済が一因 : 推論計算資源の可用性がソフトウェア生産性を支配し、推論資源やエネルギー制約がより多くのワークロードをローカルに押しやる可能性があるとの見方があります（gdb、awnihannun）。
  • Ollamaはローカルワークフローの製品化を継続 : Ollamaは0.16.3をリリースし、ollama launchで「ClineとPi統合」を提供しました（ollama）。これにより、ノートPCでOSSモデルが「ほとんどの作業に十分」な性能を発揮する時代が近いとの見方が広がっています（sdrzn）。

ハードウェア＋推論高速化：カスタムシリコン「ハードコアモデル」、ThunderKittens 2.0、スパースアテンション、高速デコード

  • Taalas「チップがモデル」極端なユーザー別スループット : 複数投稿でLlama 3 8Bがユーザーごとに約16k–17kトークン／秒を達成するデモが引用され、SRAM中心のCerebrasよりもほぼ一桁速いとされています（awnihannun、wildmindai）。Awniは実務的な反論として、テープアウトの遅延（数ヶ月）がモデル更新サイクルと合わず、ベースモデルをシリコン化＋アダプタ型後訓練のハイブリッドが現実的と述べています（awnihannun）。
  • カーネルレベルの進展 : ThunderKittens 2.0は新しいBF16/MXFP8/NVFP4 GEMMをBlackwell上でcuBLASと同等または上回る性能で実装し、「最後のTFLOPまで絞り出す」ことを強調しています（stuart_sul）。
  • 拡散／動画向けアテンションスパース化 : SpargeAttention2は95%アテンションスパース化と動画拡散で16.2倍の高速化を達成し、Top‑k＋Top‑pマスキングと蒸留ファインチューニングのハイブリッドを採用しています（HuggingPapers、_akhaliq）。

セキュリティ、ガバナンス、実世界のエージェント：Claude Code Security＋軌跡監査

  • Claude Code Security（研究プレビュー） : Anthropicはコードベースの脆弱性を検出し、人間によるレビュー用のパッチ提案を行うセキュリティスキャンエージェントを発表しました（claudeai）。続報では、OSS本番環境で500以上の脆弱性を発見し、報告・修正した例が示されています（trq212、_catwu）。第三者コードでの実行を許可しないなどの制限が「興味深い」製品選択として議論されています（moyix）。
  • エージェント軌跡の監査が新たな安全・堅牢性ツールに : Hodoscopeは軌跡を可視化・監査するツールとして紹介され、ベンチマーク脆弱性を迅速に発見したと報告されています。評価＋テレメトリーがエージェントやベンチマーク双方の欠陥を発見できることを示しています（AdtRaghunathan、gneubig）。
